# KF4 Handoff Intelligence — Design Spec

**Date:** 2026-04-03  
**Status:** Approved  
**Feature:** Killer Feature 4 — Handoff Intelligence  
**Build strategy:** 3 parallel subagents after DB migration

---

## Overview

Three sub-features that together make handoffs from design to dev visible, intelligent, and measurable:

1. **Handoff Brief** — AI-generated brief for devs at the moment of handoff
2. **Dev Questions + Quality Score** — track how many questions devs ask per designer
3. **Figma Drift Email** — notify dev owner when designer updates Figma post-handoff

---

## Sub-feature 1: Handoff Brief

### What it does
When a designer advances to the "Handoff" `designStage`, a brief is lazily generated by Claude Haiku and cached in the DB. The brief is visible to the designer at the Handoff stage, and to devs throughout the dev phase. It replaces the current blank slate devs receive.

### When it generates
On first page load when `designStage === "handoff"`. Same lazy pattern as context brief — generated on demand, cached forever, instant on repeat opens. Silent fail if AI is down.

### AI input
- Request title, description, businessContext, successMetrics
- All non-system comments (full thread)
- figmaUrl

### 5 sections
| Section | Content | Format |
|---|---|---|
| Design decisions | Key choices made and why | `[{decision, rationale}]` x2-4 |
| Open questions | Unresolved things dev should flag back | `string[]` x2-4 |
| Build sequence | Suggested dev priority order | `string[]` x2-3 |
| Figma notes | Which frames/components to look at first | `text` |
| Edge cases | Design states dev must handle | `string[]` x2-4 |

### Schema

```sql
CREATE TABLE request_handoff_briefs (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  requestId    uuid UNIQUE NOT NULL REFERENCES requests(id) ON DELETE CASCADE,
  designDecisions jsonb,
  openQuestions   jsonb,
  buildSequence   jsonb,
  figmaNotes      text,
  edgeCases       jsonb,
  aiModel         text,
  tokensUsed      integer,
  createdAt    timestamp with time zone DEFAULT now()
);
```

### Files

**Create:**
- `db/schema/handoff_briefs.ts`
- `lib/ai/handoff-brief.ts` — `generateHandoffBrief()` using Haiku + generateObject + zod
- `app/api/requests/[id]/handoff-brief/route.ts` — POST: auth + org guard + `onConflictDoNothing` + return
- `components/requests/handoff-brief-panel.tsx` — skeleton on first load, 5 sections rendered

**Modify:**
- `db/schema/index.ts` — export new table
- `app/(dashboard)/dashboard/requests/[id]/page.tsx` — fetch brief, pass to panel; render at Handoff stage in design phase and at the top of the dev phase panel (above kanban steps)
- `components/requests/design-phase-panel.tsx` — render `HandoffBriefPanel` at Handoff stage

## Sub-feature 2: Dev Questions + Quality Score

### What it does
Devs can click "Ask Designer" in the dev phase to post a tagged question. Questions are counted per-request and aggregated per-designer. Design Head sees avg questions/handoff on the Radar — high count = coaching signal.

### Dev experience
"Ask Designer" button in dev phase panel → textarea → posts comment with `isDevQuestion: true`. Question count badge shown in dev phase header ("2 dev questions").

### Quality score
- **Per-request:** raw count of `isDevQuestion` comments
- **Per-designer on Radar:** avg dev questions per handoff over last 30 days (requests that reached dev phase), color-coded: 🟢 0–1 avg, 🟡 2–3, 🔴 4+

### Schema change

```sql
ALTER TABLE comments ADD COLUMN isDevQuestion boolean NOT NULL DEFAULT false;
```

### Files

**Modify:**
- `db/schema/requests.ts` — add `isDevQuestion` to comments table Drizzle definition
- `app/api/requests/[id]/comment/route.ts` — accept `isDevQuestion` param, include in insert
- `components/requests/dev-phase-panel.tsx` — "Ask Designer" button + question count badge
- `lib/radar.ts` — add `avgDevQuestions` computation to designer rows
- `app/(dashboard)/dashboard/radar/page.tsx` — query avg dev questions per designer
- `components/radar/designer-status.tsx` — render new avg column with color coding

## Sub-feature 3: Figma Drift Email

### What it does
When a Figma update is logged with `postHandoff: true` and the request has a `devOwnerId`, send an email to the dev owner notifying them of the change. The in-app amber alert + dev review flow already exist — this adds the push notification.

### Email content
- Subject: "Figma updated on [request title]"
- Body: designer name, request link, brief note that Figma was updated post-handoff and needs dev review

### Files

**Modify:**
- `lib/email/templates.ts` — add `figmaDriftEmail()` template (alongside existing templates)

**Modify:**
- `app/api/requests/[id]/figma-updates/route.ts` — after insert, if `postHandoff: true` and `devOwnerId` set, call `sendFigmaDriftEmail()`
- `lib/email/index.ts` — add `sendFigmaDriftEmail()` function

No shared file conflicts. Run in parallel after migration.

## Key Decisions

- **Lazy generation** — handoff brief generates on first page load at Handoff stage, not on phase transition
- **Cached forever** — no regenerate button (YAGNI). Brief reflects state at first generation
- **Visible to all** — brief shown to designer + devs (not designer-only)
- **isDevQuestion is explicit** — devs must click "Ask Designer" to tag a question; general comments don't count toward score
- **Figma drift in-app is already built** — only the email push is new in KF4
- **No new Figma OAuth** — drift detection remains manual (OAuth is deferred)
