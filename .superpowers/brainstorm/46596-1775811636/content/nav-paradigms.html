A — Object-first (Linear-style)

~7 sidebar links. One powerful "Requests" view with filters, grouping (by phase, stage, assignee, project), and board/list toggle. Phases aren't separate pages — they're grouping options. Designers filter to "my work in Design," PMs filter to "my requests in Predesign."

Pros

  • Minimal sidebar, maximum power
  • One mental model for everyone
  • Scales — 10 or 100 requests, same view
  • Proven pattern (Linear, Notion, Jira)

Cons

  • Requires good filter/group UX to not feel overwhelming
  • Pipeline visibility requires active grouping
  • New users need to learn the filter system

B — Phase-first (Pipeline nav)

~10 sidebar links. Pipeline section maps directly to Lane's 4 phases. Each phase opens a dedicated board — Design shows stage columns (Sense→Prove), Dev shows kanban. The pipeline IS the navigation. Very legible for leads.

Pros

  • Pipeline is always visible in sidebar
  • Maps to Lane's core mental model
  • Leads instantly see flow
  • Counts per phase = at-a-glance health

Cons

  • More sidebar links
  • Designers mostly care about Design — other phases are noise
  • Doesn't feel as modern as Linear
  • Rigid — hard to add cross-phase views

C — Hybrid (Recommended)

~8 sidebar links. Minimal Linear-style sidebar with objects (Requests, Ideas, Cycles). But inside Requests, phase tabs across the top give you the pipeline view. Selecting "Design" tab auto-switches to stage columns. Selecting "Dev" tab shows kanban. Best of both worlds — sidebar stays clean, pipeline lives inside the Requests view.

Pros

  • Fewest sidebar links (clean)
  • Pipeline visible via tabs (discoverable)
  • Phase tabs auto-adapt view layout
  • Scales — add more objects without sidebar bloat
  • Feels modern like Linear

Cons

  • Pipeline is one click deeper than Phase-first
  • Phase tabs need good design to not feel like "just tabs"
